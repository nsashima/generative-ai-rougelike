## ⚠️ 開発上の最重要ルール

1. **日本語での対話**:
   ユーザーへの回答・解説は**必ず日本語**で行ってください。
2. **既存コメントとdocstringの維持**:
   ソースコードの変更箇所に関係のない既存のコメントやdocstringは、原則として削除・改変せずそのまま維持してください。

   コード変更後は、必ず `npm run build` を実行してTypeScriptのコンパイルおよびViteのバンドルが正常に通ることを確認してください。

## 📝 現在のゲーム仕様 (Ver 1.2.0)

### 1. 階層とボス構成
* ダンジョンは全20階層。5の倍数フロア（5F, 10F, 15F, 20F）はボスフロアであり、通路が自動的に3マス幅に拡張されます。ボスを倒すと階段（およびユニークアイテム）が出現します。
  * **5F (洞窟)**: ボス「ゴブリンキング」（ドロップ: キングの盾）
  * **10F (溶岩)**: ボス「邪教の心眼」（ドロップ: 心眼の指輪）
  * **15F (氷結)**: ボス「結晶の守護者」（ドロップ: 吹雪の魔剣）
  * **20F (宇宙)**: ボス「深淵の覇王」（ドロップ: 生成AIの秘宝 ➡ 拾うとゲームクリア）

### 2. モンスターの動的スケーリング (深度強化)
* `dungeon.ts` 内で敵が生成される際、敵テンプレートの「初登場階層」からの差分（超過階層数）に応じてステータスが自動強化されます。
  * **スケーリング係数**: 超過1階層ごとに、最大HP `+10%`、攻撃力 `+8%`、防御力 `+5%`、獲得XP `+10%`

### 3. モンスターの特有ビジュアル（スプライト定義）
* ゴブリン種・スケルトン種は、ベースのドット絵配列に新規パーツを描き加えた固有スプライトとカラーパレットが割り当てられています。
  * **ゴブリン戦士**: 鉄兜、盾/剣のドット追加（緑肌、鉄鎧）
  * **ゴブリンロード**: 黄金の王冠、ロイヤルマントのドット追加（ティール肌、紫ローブ）
  * **スケルトン戦士**: 鉄兜、錆びた鉄剣のドット追加（灰色骨、赤目）
  * **デスナイト**: 角兜、漆黒胸鎧、血塗られた大剣のドット追加（黒骨、赤目、黒鎧）
  * **フロストスケルトン**: 周囲に氷結晶ドットを散りばめたスプライト（水色骨、シアン目）
  * **ポイズンスライム**: オレンジ色（`#fb923c`）のシンボルカラーに変更。

### 4. 装備と耐久値・商人システム
* 武器・防具には耐久値が存在し、商人（ショップ）の売却リストでは `(耐久:現在値/最大値)` が表示されます。
* 装備詳細表示のショートカットキー：
  * `Alt+W`: 武器詳細、 `Alt+A`: 防具詳細、 `Alt+R`: 指輪詳細。未装備時は「〜を装備していません」と表示。

### 5. デバッグモード
* **起動**: プレイ中に `Ctrl+Shift+D` ➡ パスワード `sashima` を入力。
* **全表示テスト部屋 (展示室)**: `db-museum-btn` をクリックすると、全アイテムと全モンスターが整列した、敵のAIが動かない安全な確認用マップが生成されます。

## 🛠️ 今後の開発方針 (Next Steps)

1. **職業（ジョブ）システムの拡張**:
   * 現在 `types.ts` には `'warrior' | 'guardian' | 'ninja'` の `JobType` が定義されていますが、選択した職業による初期ステータス補正や固有スキルの影響についてさらに深掘りして実装する余地があります。
2. **さらなる階層追加（25F〜30F）**:
   * 「5の倍数階はボスフロア」の判定ロジックが `dungeon.ts` 内で `level % 5 === 0` として抽象化されているため、階層を追加するだけで通路の拡張や階段スキップは自動適用されます。新しいテーマ（例：黄金郷、熱帯雨林）やボスを定義して拡張してください。
3. **新規アイテム・特殊効果の追加**:
   * 特殊な効果を持つ指輪や、ダンジョンの壁を壊せる巻物、罠（トラップ）オブジェクトの設置などを追加することで、ローグライクとしての戦略性がより高まります。

## CHANGELOG.mdの記述

### CHANGELOG.md 更新ルール

あなたはこのリポジトリの CHANGELOG.md のメンテナです。

### 基本方針

- CHANGELOG.md はリポジトリ内で単一ファイルとして管理する
- リリースごとにファイルを分割しない
- 最新バージョンを上部に記載する
- GitHub Release の内容と同期できる形式で管理する
- Keep a Changelog の考え方を参考にする

### セクション構成

CHANGELOG.md の先頭には必ず `Unreleased` セクションを配置する。

例:

```md
# Changelog

## [Unreleased]

### Added

### Changed

### Fixed

---

## [0.5.0] - 2026-07-01

### Added
- 指輪システム追加

### Changed
- UI改善

### Fixed
- モバイル表示崩れ修正
```

## 更新ルール

### 機能追加

新機能を実装した場合:

```md
### Added
- ○○機能を追加
```

### 仕様変更

既存機能の挙動変更や改善を行った場合:

```md
### Changed
- ○○の仕様を変更
```

### バグ修正

不具合修正を行った場合:

```md
### Fixed
- ○○の不具合を修正
```

## リリース時のルール

タグ作成やリリース作成時は:

1. Unreleased の内容を新しいバージョンセクションへ移動する
2. バージョン番号とリリース日を記載する
3. 新しい空の Unreleased セクションを先頭に再作成する

例:

変更前:

```md
## [Unreleased]

### Added
- セーブ機能追加
```

変更後:

```md
## [Unreleased]

### Added

### Changed

### Fixed

---

## [0.6.0] - 2026-08-01

### Added
- セーブ機能追加
```

## 記載内容の粒度

記載する:
- ユーザーが認識できる機能追加
- UI変更
- ゲームバランス変更
- バグ修正
- パフォーマンス改善

記載しない:
- リファクタリングのみ
- コメント修正
- テストコード追加のみ
- フォーマット修正のみ

## エージェントへの指示

コード変更を行った場合は、CHANGELOG.md の Unreleased セクションに追記が必要か確認すること。

ユーザーが認識できる変更が含まれる場合は、CHANGELOG.md も合わせて更新すること。
